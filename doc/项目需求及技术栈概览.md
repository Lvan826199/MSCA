# MSCA 多设备移动端投屏控制系统技术设计方案

## 1. 概述

MSCA（Mobile Screen Control Assistant）是一个跨平台的移动设备投屏与控制工具，支持 Windows、macOS 系统以及 Web 浏览器访问。系统采用 **Python + Vue + Electron** 统一技术栈，通过分层驱动架构实现对 Android 与 iOS 设备的透明管理与并发控制。

当前设计方案涵盖两大平台：

- **Android 平台**：基于 Scrcpy 协议，通过 ADB 实现高效的 H.264 视频流传输与反向控制，支持桌面端（Electron 进程管理）与 Web 端（Python 后端转发）两种部署模式。
- **iOS 平台**：鉴于系统封闭性，采用版本适配策略，低版本（≤15.x）使用 Tidevice + WebDriverAgent，高版本（≥16.x）使用 go-ios + WebDriverAgent，实现屏幕投射与控制指令下发。

整体架构遵循统一的设备驱动抽象层，支持多设备并发管理、单页面多窗口监控及独立精准操控，无需数据库存储持久化数据。

### 2.2 核心组件职责

| 组件                     | 职责描述                                                     |
| :----------------------- | :----------------------------------------------------------- |
| **DeviceManager**        | 管理所有已连接设备（Android/iOS）的发现、注册、状态监控与移除，提供统一的设备列表 |
| **AbstractDeviceDriver** | 定义设备驱动的统一接口，包括 `start_mirroring()`、`stop_mirroring()`、`send_event()`、`get_frame()` 等，屏蔽底层平台差异 |
| **AndroidDriver**        | Android 设备驱动实现，负责调用 `ScrcpyServerManager` 推送并启动内置 scrcpy-server，处理 H.264 视频流与控制指令 |
| **ScrcpyServerManager**  | Python 后端 Android 投屏代理，通过 adbutils 与设备建立 scrcpy 协议连接，将 H.264 流通过 WebSocket 转发至前端 |
| **IOSDriver**            | iOS 设备驱动实现，负责与适配层交互，管理 WDA 服务生命周期，处理 MJPEG 视频流与控制指令 |
| **TideviceAdapter**      | 封装 Tidevice 库，提供低版本 iOS 的设备连接、WDA 启动、视频流获取与控制下发 |
| **GoIOSAdapter**         | 封装 go-ios 命令行工具，提供高版本 iOS 的设备配对、WDA 管理、视频流获取与控制下发 |
| **StreamDispatcher**     | 负责将各设备的视频流（H.264/MJPEG）分发给对应的 WebSocket 客户端 |
| **SessionManager**       | 管理每个控制会话的状态、超时、权限校验等                     |

## 4. 详细设计

### 4.1 设备发现与识别

#### 4.1.1 Android 设备发现

- **桌面端 / Web 端统一后端**：Python 后端通过 `adbutils.adb.device_list()` 获取实时设备列表，通过 WebSocket 向前端推送变更；Electron 仅负责启动和监控本地后端。

#### 4.1.2 iOS 设备发现

- **USB 连接检测**：通过监听系统 USB 事件（Windows: WMI, macOS: IOKit）或轮询 `usbmuxd` 服务获取设备插拔通知。
- **设备信息获取**：
  - Tidevice 路径：调用 `tidevice list` 获取 JSON 格式设备列表。
  - go-ios 路径：调用 `ios list` 获取设备列表。
- **版本识别**：通过设备信息中的 `ProductVersion` 字段判定 iOS 版本，自动选择适配器。

#### 4.1.3 设备注册

所有设备信息封装为统一 `DeviceInfo` 对象，包含字段：设备 ID、平台类型（Android/iOS）、系统版本、型号名称、连接状态、屏幕分辨率等，通过 WebSocket 或 IPC 推送至前端。

### 4.2 Android 设备投屏与控制实现

> **架构调整（2026-04-24）**：废弃原 4.2.1 桌面端 Electron 直接 spawn scrcpy GUI 的方案。桌面端和 Web 端统一使用 Python 后端管理 scrcpy-server，前端通过 WebSocket 接收 H.264 流并用 WebCodecs 解码渲染。详见下方 4.2.2。

#### 4.2.1 ~~桌面端方案（已废弃）~~

~~原方案通过 Electron 主进程 spawn scrcpy GUI 客户端。~~ 已废弃，原因：
- 桌面端和 Web 端实现完全不同，前端需要分支判断
- scrcpy GUI 窗口无法嵌入 Electron 渲染进程，用户体验割裂
- 控制逻辑依赖 scrcpy GUI 原生窗口事件，无法自定义

现在桌面端通过 BackendManager 自动启动本地 Python 后端（从 18000 起自动探测可用端口，实际端口写入 `.backend-port`），与 Web 端共用同一套后端逻辑，完全离线可用。

### 4.3 iOS 设备投屏与控制实现

#### 4.3.1 WebDriverAgent 部署与启动

WDA 作为运行于 iOS 设备上的 XCTest 服务，负责接收控制指令并返回屏幕数据。

**依赖条件**：

- 有效的 iOS 开发者证书（用于签名 WDA）
- 设备已信任计算机
- 开发者模式已启用（iOS 16+ 需在设置中开启）

**部署流程**：

```python
def prepare_wda(device: DeviceInfo) -> bool:
    if not is_wda_installed(device):
        adapter.install_app(device, WDA_BUNDLE_ID, signed_ipa_path)
    adapter.start_wda(device, port=8100)
    return wait_for_wda_ready(f"http://{device.ip}:8100/status")
```

**适配器差异处理**：

| 操作     | Tidevice 实现                          | go-ios 实现                                  |
| :------- | :------------------------------------- | :------------------------------------------- |
| 安装 WDA | `tidevice -u [udid] install [wda.ipa]` | `ios install --path=[wda.ipa] --udid=[udid]` |
| 启动 WDA | `tidevice -u [udid] wdaproxy -p 8100`  | `ios runwda --udid=[udid] --port=8100`       |
| 端口转发 | 自动建立 usbmuxd 隧道                  | 需配合 `ios tunnel` 命令                     |

#### 4.3.2 视频流获取

iOS 设备视频流通过 WDA 的 MJPEG 流式接口获取。

- **MJPEG 流地址**：通过独立 MJPEG 端口读取（设备端默认 9100，本地端口由后端自动分配，独立于 WDA API 端口）
- **数据格式**：`multipart/x-mixed-replace` 中的 JPEG 图像帧

**流处理流程**：

1. 适配器分别建立 WDA API 与 MJPEG 流端口转发（设备端默认 8100/9100，本地端口由后端自动分配）。
2. `IOSDriver` 通过 HTTP 客户端持续读取 `/stream` 响应体。
3. 解析 `multipart` 边界，提取每帧 JPEG 数据。
4. 将 JPEG 二进制数据通过 WebSocket 推送给前端。
5. 前端使用 `<img>` 标签或 Canvas 渲染，亦可使用 WebCodecs 进行 JPEG 解码渲染。

**性能优化**：开启 WDA 流缩放参数，降低分辨率以提高帧率：`/stream?scale=50`

#### 4.3.3 控制指令下发

控制指令通过 WDA 的 `/session` REST API 下发。

**支持的操作类型**：

| 操作类型     | WDA API 端点                                  | 参数示例                                                     |
| :----------- | :-------------------------------------------- | :----------------------------------------------------------- |
| 点击         | `POST /session/:id/wda/tap/0`                 | `{ "x": 100, "y": 200 }`                                     |
| 双击         | `POST /session/:id/wda/doubleTap`             | `{ "x": 100, "y": 200 }`                                     |
| 长按         | `POST /session/:id/wda/touchAndHold`          | `{ "x": 100, "y": 200, "duration": 1.0 }`                    |
| 滑动         | `POST /session/:id/wda/touch/down` → `move` → `up` | `{ "x": 100, "y": 200 }` |
| 兼容拖拽     | `POST /session/:id/wda/dragfromtoforduration` | `{ "fromX": 0, "fromY": 0, "toX": 100, "toY": 100, "duration": 0.5 }` |
| 按键（Home） | `POST /session/:id/wda/pressButton`           | `{ "name": "home" }`                                         |
| 文本输入     | `POST /session/:id/element/:eid/value`        | `{ "value": ["text"] }`                                      |

**坐标适配**：前端捕获的鼠标事件坐标需转换为设备物理分辨率坐标，由驱动层维护设备尺寸并进行比例换算。

### 4.4 多设备并发管理

#### 4.4.1 Android 并发

- **统一方案**：后端为每个设备创建独立的 `ScrcpyProxy` 实例，通过独立的 WebSocket 连接与前端通信。
- **前端 UI**：MirrorView 单页面网格布局，每台设备对应一个 `DeviceMirrorPanel` 组件（独立 Canvas + WebSocket 连接），网格自适应列数。
- **同步操作**：MirrorView 提供"同步模式"开关，开启后触控/按键事件广播至所有当前投屏设备的 control_socket。

#### 4.4.2 iOS 并发

每个 iOS 设备需分配独立的 WDA 服务端口，避免端口冲突。

- **端口分配策略**：以 8100 为基础，每个设备递增 10（如 8100、8110、8120）。
- **进程隔离**：每个设备的端口转发进程由 `IOSDriver` 作为子进程管理，设备断开时终止对应进程。

### 4.5 平台统一抽象

通过继承 `AbstractDeviceDriver` 实现 `AndroidDriver` 与 `IOSDriver`，对外暴露统一接口：

```python
class AbstractDeviceDriver(ABC):
    @abstractmethod
    async def start_mirroring(self, options: MirrorOptions) -> str: ...
    
    @abstractmethod
    async def stop_mirroring(self) -> None: ...
    
    @abstractmethod
    async def send_event(self, event: ControlEvent) -> bool: ...
    
    @abstractmethod
    async def get_screenshot(self) -> bytes: ...
```

上层调度逻辑无需关心设备类型，前端统一展示和控制。

## 5. 工作流程

### 5.1 Android 设备投屏启动流程（桌面端）

1. 用户通过 USB 连接 Android 设备，开启 USB 调试。
2. Electron 主进程通过 ADB 检测到设备，推送至前端设备列表。
3. 用户点击“开始投屏”，主进程接收配置参数。
4. 主进程通过 `child_process.spawn` 启动 scrcpy 子进程。
5. scrcpy 进程自动创建投屏窗口并开始镜像与控制。

### 5.2 Android 设备投屏启动流程（统一方案）

> **UI 方案**：采用单页面多设备网格布局，路由为 `/mirror`，不弹出独立窗口。

1. Python 后端通过 adbutils 检测到设备，通过 WebSocket 推送列表至前端。
2. 用户在设备列表页（`/`）点击设备卡片的"投屏"按钮。
3. 前端路由跳转至 `/mirror?device=<id>`，进入 MirrorView 多设备网格页面。
4. MirrorView 向后端发起 `POST /api/mirror/{device_id}/start`，后端启动 scrcpy-server，建立视频与控制通道。
5. 前端建立 WebSocket 连接至 `/ws/video/{device_id}`，后端持续推送 H.264 帧。
6. 前端使用 WebCodecs 解码 H.264 并渲染至对应设备的 Canvas。
7. 用户可在 MirrorView 中继续添加其他设备，网格自动扩展。
8. 开启"同步模式"后，触控/按键操作通过 `/ws/control/{device_id}` 广播至所有当前投屏设备。
9. 前端捕获 Canvas 上的鼠标/触摸事件，坐标转换后通过 WebSocket 回传后端，后端写入 control_socket。

### 5.3 iOS 设备投屏启动流程

1. 用户连接 iOS 设备至计算机（USB 或 WiFi）。
2. DeviceManager 检测到设备插入，获取设备信息，根据 iOS 版本创建适配器实例。
3. 适配器检查并安装/启动 WebDriverAgent。
4. 适配器建立本地端口转发（8100 + offset）。
5. IOSDriver 实例化，状态变更为 Ready。
6. 用户点击“开始投屏”。
7. IOSDriver 启动视频流读取循环，将 MJPEG 帧推入 StreamDispatcher。
8. 前端通过 WebSocket 接收视频帧并渲染。
9. 用户操作触发控制指令，经 WebSocket 发送至 IOSDriver。
10. IOSDriver 调用适配器将指令转为 WDA HTTP 请求下发。

### 5.4 设备断开处理流程

1. DeviceManager 检测到设备移除信号。
2. 通知对应 Driver 停止投屏与控制。
3. 适配层终止相关子进程或端口转发。
4. 释放设备资源，从设备列表移除。
5. 前端收到设备离线通知，移除对应投屏窗口。

## 8. ws-scrcpy 协议参考（核心实现借鉴）

> ws-scrcpy 是目前最完整的 Web 端 scrcpy 协议实现，MSCA 的 Android 投屏控制核心逻辑重度参考该项目。
> 仓库地址：https://github.com/NetrisTV/ws-scrcpy

### 8.1 架构差异对比

| 维度 | ws-scrcpy | MSCA |
| :--- | :--- | :--- |
| 后端语言 | TypeScript (Node.js) | Python (FastAPI) |
| scrcpy-server 通信 | 修改版 server 内置 WebSocket | 标准 server + Python 代理转发 |
| 前端解码 | 多解码器可选（WebCodecs / Broadway / TinyH264 / MSE） | WebCodecs 优先，MSE 降级 |
| 控制消息传输 | 二进制 WebSocket（前端直接编码） | JSON WebSocket（后端转二进制） |
| 多路复用 | Multiplexer 单连接多流 | 每设备独立 WebSocket |

### 8.2 scrcpy 控制消息二进制协议

所有控制消息第一个字节为 `type`，后续为指令特定数据，大端序。

#### 控制消息类型枚举

```
TYPE_INJECT_KEYCODE              = 0
TYPE_INJECT_TEXT                 = 1
TYPE_INJECT_TOUCH                = 2
TYPE_INJECT_SCROLL               = 3
TYPE_BACK_OR_SCREEN_ON           = 4
TYPE_EXPAND_NOTIFICATION_PANEL   = 5
TYPE_EXPAND_SETTINGS_PANEL       = 6
TYPE_COLLAPSE_PANELS             = 7
TYPE_GET_CLIPBOARD               = 8
TYPE_SET_CLIPBOARD               = 9
TYPE_SET_SCREEN_POWER_MODE       = 10
TYPE_ROTATE_DEVICE               = 11
```

#### 触控消息 (TYPE=2) — 32 字节

```
偏移  长度  类型        字段
0     1    UInt8       type (2)
1     1    UInt8       action (DOWN=0, UP=1, MOVE=2)
2     8    Int64BE     pointerId (-1 表示鼠标)
10    4    Int32BE     position.x
14    4    Int32BE     position.y
18    2    UInt16BE    position.screenWidth
20    2    UInt16BE    position.screenHeight
22    2    UInt16BE    pressure (0~0xFFFF, 浏览器 force * 65535)
24    4    UInt32BE    actionButton
28    4    UInt32BE    buttons
```

MSCA 后端对应实现：`protocol.encode_inject_touch()` 使用 `struct.pack(">BBqiiHHHII", ...)`

#### 按键消息 (TYPE=0) — 14 字节

```
偏移  长度  类型        字段
0     1    UInt8       type (0)
1     1    UInt8       action (DOWN=0, UP=1)
2     4    Int32BE     keycode (Android KeyEvent 常量)
6     4    Int32BE     repeat
10    4    Int32BE     metaState (修饰键位掩码)
```

metaState 组合：Alt=2, Shift=1, Ctrl=4096, Meta=65536, CapsLock=1048576

#### 文本注入消息 (TYPE=1) — 变长

```
偏移  长度      类型        字段
0     1        UInt8       type (1)
1     4        UInt32BE    textLength (UTF-8 字节长度)
5     N        Bytes       text (UTF-8 编码)
```

#### 滚动消息 (TYPE=3) — 25 字节

```
偏移  长度  类型        字段
0     1    UInt8       type (3)
1     4    Int32BE     position.x
5     4    Int32BE     position.y
9     2    UInt16BE    screenWidth
11    2    UInt16BE    screenHeight
13    4    Int32BE     hScroll
17    4    Int32BE     vScroll
21    4    UInt32BE    buttons
```

ws-scrcpy 对滚动事件有 30ms 节流，同方向连续滚动会被合并。

#### 剪贴板设置 (TYPE=9) — 变长

```
偏移  长度      字段
0     1        type (9)
1     1        paste (0 或 1，是否同时粘贴)
2     4        textLength (Int32BE)
6     N        text (UTF-8)
```

#### 屏幕电源模式 (TYPE=10) — 2 字节

```
偏移  长度  字段
0     1    type (10)
1     1    mode (0=OFF, 2=NORMAL)
```

#### 旋转设备 (TYPE=11) — 1 字节

```
偏移  长度  字段
0     1    type (11)
```

### 8.3 视频流协议

#### 初始连接信息

ws-scrcpy 使用 magic bytes 区分消息类型：

| Magic Bytes | 含义 |
| :--- | :--- |
| `scrcpy_initial` (14字节) | 初始设备信息（设备名、显示器列表、编码器列表） |
| `scrcpy_message` (14字节) | 设备消息（剪贴板内容、文件推送响应） |
| 其他 | H.264 视频帧数据 |

初始信息结构：
```
14 bytes   magic "scrcpy_initial"
64 bytes   deviceName (UTF-8, 零填充)
4  bytes   displaysCount (Int32BE)
变长       每个 display 的 DisplayInfo (24字节) + ScreenInfo (25字节) + VideoSettings
4  bytes   encodersCount → 编码器名称列表
4  bytes   clientId
```

MSCA 的差异：MSCA 后端自行解析 scrcpy-server 的初始信息（设备名 64 字节 + 分辨率 4 字节），通过 JSON config 消息 `{ type: "config", width, height }` 发送给前端，不使用 magic bytes 机制。

#### H.264 帧格式

scrcpy-server 发送 Annex B 格式的 H.264 裸流（`0x00000001` start code 前缀），每帧可能包含多个 NAL 单元：

| NAL 类型 | 值 | 说明 |
| :--- | :---: | :--- |
| SPS | 7 | 序列参数集，包含 profile/level/分辨率信息 |
| PPS | 8 | 图像参数集 |
| IDR | 5 | 关键帧（I 帧） |
| non-IDR | 1 | 非关键帧（P/B 帧） |
| SEI | 6 | 补充增强信息 |
| AUD | 9 | 访问单元分隔符 |

MSCA 后端在转发时添加 1 字节帧类型前缀：`0x01` = key frame, `0x00` = delta frame。

### 8.4 WebCodecs 解码实现参考

ws-scrcpy 的 `WebCodecsPlayer` 实现是 MSCA 前端解码的核心参考：

#### codec 字符串构造

从 SPS NAL 的第 2~4 字节构造：`avc1.{profile_idc}{constraint_set_flags}{level_idc}`（十六进制）

```javascript
// 示例：SPS[1]=0x64, SPS[2]=0x00, SPS[3]=0x28 → "avc1.640028"
const codec = `avc1.${hex(sps[1])}${hex(sps[2])}${hex(sps[3])}`
```

#### 解码器配置

```javascript
const config = {
  codec: "avc1.640028",        // 从 SPS 解析
  optimizeForLatency: true,     // 低延迟优化
  description: avcConfigRecord, // AVCDecoderConfigurationRecord (可选)
}
decoder.configure(config)
```

MSCA 当前实现了两种模式：
- AVC 模式（有 description）：构建 AVCDecoderConfigurationRecord，帧数据转为 4 字节长度前缀格式
- Annex B 模式（无 description）：直接送入原始数据

#### 帧渲染

```javascript
output: (frame) => {
  ctx.drawImage(frame, 0, 0)
  frame.close()  // 必须释放 VideoFrame
}
```

### 8.5 触控坐标转换

ws-scrcpy 的坐标转换逻辑（`InteractionHandler.buildTouchOnClient`）：

```
浏览器坐标 → Canvas 相对坐标 → 设备物理坐标

scaleX = videoWidth / canvasDisplayWidth
scaleY = videoHeight / canvasDisplayHeight
deviceX = (clientX - canvasRect.left) * scaleX
deviceY = (clientY - canvasRect.top) * scaleY
```

MSCA 的 `useDeviceControl.canvasToDevice()` 已实现相同逻辑。

## 9. 打包部署架构

### 9.1 混合部署模式

MSCA 采用混合部署架构，同一套前端代码同时支持桌面端（内嵌后端）和纯 Web 端（远程后端）两种使用方式：

- **桌面端**：用户双击应用后，Electron 主进程自动启动内嵌的 Python 后端（Nuitka 编译产物），从 18000 起自动探测本地可用端口并写入 `.backend-port`，完全离线可用。
- **Web 端**：部署公共后端服务（用户自有服务器），浏览器通过 HTTPS/WSS 连接远程后端，实现远程投屏控制。

```
┌─────────────────────────────────────────────────────────────┐
│                      桌面端（Electron）                       │
│  ┌──────────────┐    IPC     ┌──────────────────────────┐   │
│  │  主进程        │◄────────►│  BackendManager           │   │
│  │  (main.js)    │           │  启动/监控/重启后端进程    │   │
│  └──────────────┘           └──────────┬───────────────┘   │
│  ┌──────────────┐                      │ spawn              │
│  │  渲染进程      │    WebSocket        ▼                    │
│  │  (Vue 3)      │◄──────────► msca-backend.exe             │
│  │              │  ws://127.0.0.1:<.backend-port>  (Nuitka 产物) │
│  └──────────────┘                                           │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                      Web 端（浏览器）                         │
│  ┌──────────────┐   HTTPS/WSS   ┌────────────────────────┐ │
│  │  Vue 3 前端   │◄────────────►│  远程 Python 后端        │ │
│  │  (静态部署)   │              │  (Nginx + FastAPI)      │ │
│  └──────────────┘              └────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

### 9.2 后端打包方案

| 项目 | 说明 |
| :--- | :--- |
| **打包工具** | Nuitka（Python → C 编译，生成独立可执行文件） |
| **目标平台** | Windows（.exe），macOS 暂不支持 |
| **体积限制** | 不设硬性限制，以功能完整为优先 |
| **包含依赖** | FastAPI、uvicorn、adbutils、websockets 及 Python 运行时 |
| **健康检查** | 提供 `GET /health` 端点，返回 200 表示服务就绪 |

### 9.3 Electron 后端进程管理（BackendManager）

Electron 主进程内置 `BackendManager` 模块，负责本地后端进程的完整生命周期：

| 职责 | 说明 |
| :--- | :--- |
| **启动** | 应用启动时检测后端可执行文件，spawn 子进程 |
| **健康检查** | 轮询 `GET /health`，确认后端就绪后通知渲染进程 |
| **崩溃重启** | 自动重启，最多 3 次，间隔递增（立即 → 3s → 5s），超限后弹窗提示用户 |
| **优雅关闭** | 应用退出时发送终止信号，等待后端进程退出 |

### 9.4 前端连接模式

前端通过 `window.electronAPI` 判断运行环境，支持三种连接模式：

| 模式 | 说明 | 适用场景 |
| :--- | :--- | :--- |
| **自动（auto）** | 桌面端默认。优先连接本地后端，失败则提示配置远程地址 | 桌面端日常使用 |
| **仅本地（local）** | 仅连接本地后端实际端口（从 `.backend-port` / Electron API 获取），不尝试远程 | 离线环境 |
| **仅远程（remote）** | 连接用户配置的远程后端地址（WSS） | Web 端、远程控制场景 |

- 桌面端（`isElectron = true`）：默认 auto 模式，可在设置中切换。
- Web 端（`isElectron = false`）：仅 remote 模式，需配置或输入远程后端地址。

### 9.5 端口管理策略

| 端口号 | 用途 | 处理方式 |
| :--- | :--- | :--- |
| **18000** | 默认本地后端端口 | 启动前检测可用性，若被占用则尝试 18001、18002 |
| **8100~** | iOS WDA 端口转发 | 每设备递增 10（8100、8110、8120），由后端管理 |

### 9.6 Web 端部署方案

Web 端部署推荐使用 Nginx 反向代理处理 TLS 加密：

```
用户浏览器 ──(HTTPS/WSS)──> Nginx ──(HTTP/WS)──> Python FastAPI 后端
```

- Nginx 负责 TLS 终止、静态文件托管（Vue 构建产物）、WebSocket 代理。
- Python 后端仅监听 HTTP/WS，无需处理证书。

### 9.7 构建产物

| 产物 | 内容 | 用途 |
| :--- | :--- | :--- |
| **MSCA-Setup.exe** | Electron 应用 + Vue 前端 + msca-backend.exe + scrcpy 二进制 | Windows 桌面端安装包 |
| **msca-web/** | Vue 静态文件（HTML/JS/CSS） | Web 端部署至 Nginx 等 Web 服务器 |
| **msca-backend.exe** | Nuitka 编译的 Python 后端 | 桌面端内嵌 / Web 端独立部署 |

## 10. 后续扩展方向

1. **macOS 桌面端支持**：Nuitka 编译 macOS 可执行文件，配合 Apple Developer 证书签名与公证（Notarization），实现 macOS 桌面端分发。
2. **iOS H.264 硬编码流支持**：利用 go-ios 的 `record` 命令获取 H.264 视频流，通过 WebCodecs 解码，大幅降低带宽占用并提升流畅度。
3. **无线调试支持**：Android 支持 ADB over TCP/IP，iOS 基于 go-ios 的 WiFi 连接能力，实现脱离 USB 的远程控制。
4. **屏幕录制与回放**：集成视频流保存功能，用于测试回放分析。
5. **多设备同步操作**：支持将同一控制指令广播至多台设备，用于批量测试与演示。
6. **插件化扩展**：设计插件体系，允许第三方开发者扩展设备类型（如鸿蒙设备）或增加自定义控制面板。

本方案为 MSCA 提供了完整的跨平台移动设备投屏控制技术框架。Android 部分可快速基于 Escrcpy 或 ws-scrcpy 进行二次开发实现；iOS 部分建议第一阶段聚焦于 Tidevice 适配器实现基础投屏控制，第二阶段引入 go-ios 覆盖高版本设备并优化性能。通过统一的驱动抽象层，两大平台可无缝融入同一套前后端架构，最终实现真正的多平台移动设备集中管控解决方案。
