### M3/M5 Android 统一投屏（Python 后端 + WebCodecs）

**目标**：Python 后端管理 scrcpy-server，通过 WebSocket 转发 H.264 流，前端 WebCodecs 解码渲染至 Canvas。桌面端和 Web 端共用同一套实现。

> **ws-scrcpy 参考**：视频流协议格式、H.264 NAL 解析、WebCodecs 解码器配置（SPS 解析 → codec 字符串 → AVCDecoderConfigurationRecord）均参考 ws-scrcpy 的 `WebCodecsPlayer` 和 `StreamReceiver` 实现。详见技术栈概览 §8.3 和 §8.4。

**UI 方案**：单页面多设备网格布局（`/mirror` 路由），不弹出独立窗口。点击设备卡片"投屏"按钮跳转至 MirrorView，支持在同一页面追加多台设备并发投屏，为后续多设备同步操作（M7）奠定基础。

| 子任务 | 描述 | 验收标准 |
| :--- | :--- | :--- |
| M3/M5.1 ScrcpyServerManager | 推送 scrcpy-server jar、启动 server、建立 video/control socket | 设备端 scrcpy-server 进程运行 |
| M3/M5.2 scrcpy 协议解析 | H.264 NAL 解析、控制指令编码 | 指令格式符合 scrcpy 协议规范 |
| M3/M5.3 投屏 REST API | `POST /api/mirror/{device_id}/start`、`stop` | 可启动/停止指定设备投屏 |
| M3/M5.4 视频流 WS 端点 | `/ws/video/{device_id}` 推送 H.264 帧 | 前端收到完整的 H.264 帧数据 |
| M3/M5.5 AndroidDriver 实现 | 实现 AbstractDeviceDriver，调用 ScrcpyServerManager | 驱动接口统一，可被上层调度 |
| M3/M5.6 WebCodecs 前端解码 | H.264 解码 + Canvas 渲染（降级方案：MSE） | 浏览器中显示设备实时画面 |

### M4/M6 Android 统一控制（前端事件 → WS → 后端）

**目标**：前端捕获触控/按键事件，通过 WebSocket 发送至后端，后端编码为 scrcpy 二进制协议写入 control_socket。

> **ws-scrcpy 参考**：控制消息的二进制编码格式直接参考 ws-scrcpy 的 `ControlMessage` 实现（详见技术栈概览 §8.2）。MSCA 采用前端 JSON → 后端转二进制的方式，降低前端复杂度。

| 子任务 | 描述 | 验收标准 |
| :--- | :--- | :--- |
| M4/M6.1 控制指令 WS 端点 | `/ws/control/{device_id}` 接收 JSON 指令，后端编码为 scrcpy 二进制 | 指令正确写入 control_socket |
| M4/M6.2 触控事件捕获与坐标转换 | Canvas 鼠标/触摸事件 → 设备坐标 → WS 发送（参考 ws-scrcpy InteractionHandler） | 点击/滑动操作正确映射到设备 |
| M4/M6.3 控制栏组件 | Home/Back/电源/音量/通知栏/最近任务按钮 | 虚拟按键响应正确 |
| M4/M6.4 键盘事件映射 | 浏览器 KeyboardEvent → Android keycode（参考 ws-scrcpy KeyToCodeMap） | 物理键盘按键正确传递到设备 |
| M4/M6.5 文本输入 | TYPE_INJECT_TEXT 注入 UTF-8 文本 | 可输入中英文文本 |
| M4/M6.6 剪贴板同步 | TYPE_GET_CLIPBOARD + TYPE_SET_CLIPBOARD 双向同步 | 可在设备与电脑间复制粘贴 |
| M4/M6.7 屏幕旋转自适应 | 监听分辨率变化，Canvas 自动调整比例 | 设备旋转后画面比例正确 |
| M4/M6.8 滚动事件 | 鼠标滚轮 → TYPE_INJECT_SCROLL，30ms 节流 | 滚动操作流畅 |

### M7 多设备并发管理与同屏监控

>
> **ws-scrcpy 参考**：ws-scrcpy 使用 Multiplexer 在单 WebSocket 连接上多路复用多设备流，减少连接数。MSCA 当前采用每设备独立 WebSocket 的简单方案，如后续遇到连接数瓶颈可参考其多路复用设计。

## 4. 可行性分析补充

### 4.1 技术风险与应对

| 风险项 | 影响等级 | 应对策略 |
| :--- | :---: | :--- |
| scrcpy-server 协议变更 | 中 | 锁定 scrcpy 版本（≥2.0），建立协议版本检测机制 |
| WebCodecs API 浏览器兼容性 | 中 | 实现 MSE 降级方案，运行时检测 API 可用性 |
| WDA 签名 7 天过期 | 高 | 文档明确要求付费开发者账号；提供重签脚本 |
| iOS 系统更新导致 go-ios 不兼容 | 中 | 关注 go-ios 项目更新，建立版本探测回退机制 |
| MJPEG 流延迟较高 | 中 | 支持 `?scale=50` 降低分辨率；后续探索 H.264 流 |
| 多设备并发时内存压力 | 中 | 实现背压控制，限制最大并发设备数 |

### 4.2 技术验证优先项

在正式开发前，建议对以下技术点进行原型验证：

1. **Python scrcpy 协议代理**：验证通过 adbutils 建立 scrcpy 连接并转发 H.264 流的可行性
2. **WebCodecs H.264 解码**：验证浏览器端 H.264 裸流解码与 Canvas 渲染的性能
3. **Tidevice WDA 管理**：验证 WDA 安装、启动、MJPEG 流获取的完整链路
4. **go-ios 高版本适配**：验证 iOS 16+ 设备的配对与 WDA 部署流程
