## 二、各项修复说明(给阅读者的解释)

### 1. iOS 触控坐标偏移 / 无法点击(第五轮)

**现象**:iOS 17.7.4 滑动正常但无法点击;iOS 15.1 触控位置与页面响应位置不一致;返回、音量等快捷键正常。

**根因(两台设备同一个 bug)**:前端发给后端的是 MJPEG 帧的**像素坐标**,后端 iOS 驱动把它原样写进了 WDA 的 W3C Actions——但 WDA 坐标系是**点(逻辑分辨率)**。iPhone 有 2x/3x 缩放(像素 = 点 × scale),所以坐标被系统性放大了 2~3 倍:

- 15.1:点击/滑动位置整体偏移(只有屏幕左上 1/scale 区域能映射回屏幕内)
- 17.7.4:点击落点越界被 WDA 忽略 → "无法点击";滑动被钳制后相对位移仍能滚动 → 看似"滑动正常"
- 快捷键走的是 `pressButton`/`homescreen` 等不带坐标的端点,所以不受影响——这是定位的关键线索

**修复**:投屏启动时同时获取窗口点尺寸(`/window/size`)和帧像素尺寸,所有 tap/swipe 坐标按比例换算成点并钳制在屏幕内;前端随事件携带的帧尺寸透传到驱动,屏幕旋转后依然跟手。

### 2. 后端运行日志查看(第五轮)

侧边栏新增**"运行日志"**页面:自动刷新(2 秒,可关)、底部跟随滚动、一键复制,桌面端与 Web 端通用。后端日志同时落盘:打包版在 `%APPDATA%\MSCA\logs\backend.log`(5MB×3 滚动),页面顶部显示具体路径。iOS 投屏启动时会打印 `窗口 375x812 pt, 帧 1125x2436 px (scale≈3.00)` 这样的诊断日志。

### 3. 关闭应用后子进程残留(第六轮)

**现象**:关闭 MSCA 后 `ios.exe`、`msca-backend.exe` 残留,导致下次打包时 electron-builder 清理 `win-unpacked` 被文件占用拒绝。

**两层根因**:① go-ios tunnel agent 是主机级共享守护进程,设备停止投屏时刻意不杀它,但应用关闭路径上没有任何代码负责终止它 → 必然残留;② Windows 上 Electron 的 `proc.kill("SIGTERM")` 等价于强杀,只杀后端自己、不杀子进程树,且后端来不及做优雅清理 → 子进程全变孤儿。

**修复(双保险)**:后端 lifespan 关闭时统一终止登记过的 tunnel agent;Windows 下 Electron 改用 `taskkill /PID <pid> /T /F` 整树强杀兜底。

### 4. 启动时清理残留进程(第七轮,用户提议的方案)

在第六轮"正常关闭时清理"之外,补上"上次异常退出(崩溃/任务管理器强杀)留下的残留":`BackendManager.start()` 启动后端前先清理——残留 `msca-backend.exe` 按映像名整树强杀;`ios.exe` 名字太通用,改用 PowerShell **按可执行文件路径过滤**,只杀本应用 `bin/ios` 目录下启动的进程,避免误伤无关进程。同时给 Electron 加了**单实例锁**,防止双开 MSCA 时清理逻辑互杀对方的后端。

### 5. go-ios 升级 v1.0.211 → v1.2.0(第七轮)

最新版 v1.2.0(2026-06-08 发布)。适配器用到的全部子命令(list/apps/tunnel/info/forward/runwda/install/image)已在 v1.2.0 逐一核对兼容。Windows 版 `ios.exe` 已入库,Linux 版留本机验证用(已 gitignore)。

### 6. wintun.dll 内嵌(第七轮,采用打包内嵌方案并有更优发现)

**背景**:go-ios 内核 tunnel 依赖 wintun.dll,项目原先未打包也未在文档说明来源,用户缺失时 iOS 17+ 设备会连续失败被标记 unavailable。

**关键发现**:go-ios 的 wintun 加载器带 `LOAD_LIBRARY_SEARCH_APPLICATION_DIR` 标志——**wintun.dll 放在 ios.exe 同目录就能被找到,根本不需要复制到 System32**。所以内嵌 `bin/ios/wintun.dll`(0.14.1 amd64)后打包零配置生效。另外 MSCA 首选的 `--userspace` 模式本来就不依赖 wintun,wintun 只影响提权内核回退路径。许可合规:wintun 再分发条款允许"随仅通过其 API 使用它的软件"分发,已附 `bin/ios/wintun-LICENSE.txt`。

### 7. 发包安全性评估(第七轮)

**结论:风险可接受,不建议投入加壳混淆。** 后端是 Nuitka 编译的机器码,不存在 PyInstaller 那种"一键解包还原 .py"的途径,防普通用户提取源码足够。前端 Electron 的 asar 一条命令就能解包看到 minified JS,但项目没有服务端密钥、没有付费授权算法,核心协议(scrcpy/WDA)本身开源,被看到的实际损失很低。**真正要守住的**是别把配对凭证(`selfIdentity.plist`)、证书打进安装包——已确认在 .gitignore 且 extraResources 不包含。
