## 1. 开发环境搭建

### 1.1 前置依赖

| 工具 | 版本要求 | 用途 |
| :--- | :--- | :--- |
| Node.js | ≥18.x | 前端构建与 Electron 运行 |
| Python | ≥3.13 | 后端服务 |
| uv | 最新版 | Python 包管理与虚拟环境 |
| ADB (Platform Tools) | ≥33.0 | Android 设备通信 |
| Git | 最新版 | 版本控制 |

### 1.2 安装 Node.js

1. 访问 https://nodejs.org/ 下载 LTS 版本
2. 安装时勾选"Add to PATH"
3. 验证：`node --version` 和 `npm --version`

### 1.3 安装 Python 与 uv

1. 访问 https://www.python.org/downloads/ 下载 Python 3.13+
2. 安装时勾选"Add Python to PATH"
3. 安装 uv：

```bash
# Windows (PowerShell)
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
```

4. 验证：`python --version` 和 `uv --version`

### 1.4 安装 ADB (Android Platform Tools)

ADB 是与 Android 设备通信的核心工具，需要单独安装。

#### Windows

1. 访问 https://developer.android.com/tools/releases/platform-tools 下载 Windows 版本
2. 解压到固定目录，例如 `D:\android_sdk\platform-tools`
3. 将该目录添加到系统环境变量 PATH：
   - 右键"此电脑" → 属性 → 高级系统设置 → 环境变量
   - 在"系统变量"中找到 `Path`，点击编辑
   - 新建一行，填入 `D:\android_sdk\platform-tools`
   - 确定保存
4. 重新打开终端，验证：`adb version`

#### macOS

```bash
# 方式一：Homebrew
brew install android-platform-tools

# 方式二：手动下载
# 访问上述链接下载 macOS 版本，解压后将路径加入 ~/.zshrc：
# export PATH="$PATH:/path/to/platform-tools"
```

验证：`adb version`，应输出类似 `Android Debug Bridge version 1.0.41`。

#### USB 驱动（Windows）

部分 Android 设备在 Windows 上需要安装 USB 驱动：

- 小米/Redmi：安装小米助手或从设备管理器更新驱动
- 三星：安装 Samsung USB Driver
- 通用方案：安装 Google USB Driver（通过 Android SDK Manager）

如果 `adb devices` 显示设备但状态为 `unauthorized` 或 `offline`，通常是驱动问题。

## 2. Android 设备连接

### 2.1 开启 USB 调试

1. 进入手机"设置" → "关于手机"
2. 连续点击"版本号"7 次，开启开发者选项
3. 返回"设置" → "开发者选项"（部分手机在"更多设置"中）
4. 开启以下选项：
   - **USB 调试**：必须开启
   - **USB 安装**：建议开启（部分操作需要）
   - **USB 调试（安全设置）**：小米设备建议开启

### 2.2 连接设备

1. 使用 USB 数据线连接手机与电脑
2. 手机弹出"允许 USB 调试"对话框，勾选"始终允许"并确认
3. 验证连接：

```bash
adb devices
```

正常输出示例：

```
List of devices attached
e63590450406    device
```

状态说明：

| 状态 | 含义 | 处理方式 |
| :--- | :--- | :--- |
| `device` | 正常连接 | 无需处理 |
| `unauthorized` | 未授权 | 在手机上确认 USB 调试授权弹窗 |
| `offline` | 设备离线 | 重新插拔 USB 或执行 `adb kill-server && adb devices` |
| 无设备显示 | 未识别 | 检查 USB 线、驱动、USB 调试开关 |

### 2.3 获取设备信息

```bash
# 设备型号
adb shell getprop ro.product.model

# Android 版本
adb shell getprop ro.build.version.release

# 屏幕分辨率
adb shell wm size
```

## 4. 启动开发环境

### 4.1 端口管理机制

后端采用动态端口发现机制，不再使用固定端口：

1. 后端启动时从 18000 开始自动探测可用端口（18000 → 18001 → 18002...）
2. 实际端口写入项目根目录 `.backend-port` 文件
3. 前端 Vite 开发服务器读取该文件，自动配置 proxy 转发
4. 后端退出时自动清理 `.backend-port` 文件

> 因此，前端无需关心后端端口号，所有 API 和 WebSocket 请求通过 Vite proxy 透明转发。

### 4.2 桌面端开发（Electron，推荐）

```bash
npm run electron:dev
```

该命令会自动完成以下操作，无需手动干预：
1. 启动 Python 后端服务（自动分配端口，写入 `.backend-port`）
2. 启动 Vite 前端开发服务器（localhost:5173，自动配置 proxy）
3. 等待前端就绪后打开 Electron 桌面窗口

> 这是最简单的启动方式，适合日常开发。

### 4.3 Web 端开发（浏览器）

Web 端开发需要分两步启动，**顺序不能颠倒**：

```bash
# 终端 1：先启动后端（自动找端口，写入 .backend-port）
npm run dev:backend

# 终端 2：再启动前端（读取 .backend-port 配置 proxy）
npm run dev
```

浏览器访问 http://localhost:5173 查看界面。

> 如果先启动前端再启动后端，Vite 读不到端口文件会 fallback 到 18000。此时需重启前端。

### 4.4 单独启动后端（调试用）

```bash
cd backend
uv run python __main__.py
```

后端启动后会输出实际端口号：

```
[backend] 启动于端口 18000，端口文件: E:\...\MSCA\.backend-port
```

验证：访问 `http://127.0.0.1:<实际端口>/health`，应返回 `{"status":"ok"}`。

也可以手动指定端口：

```bash
uv run python __main__.py --port 19000
```

### 4.5 开发模式选择

| 场景 | 启动方式 | 说明 |
| :--- | :--- | :--- |
| 桌面端开发（推荐） | `npm run electron:dev` | 一键启动，自动管理前后端 |
| Web 端开发 | 终端 1: `npm run dev:backend` → 终端 2: `npm run dev` | 浏览器访问，热更新快 |
| 后端调试 | `npm run dev:backend` 或 `cd backend && uv run python __main__.py` | 单独调试 API 和 WebSocket |

### 4.6 开发环境验证

启动后检查以下几项确认环境正常：

| 检查项 | 预期结果 |
| :--- | :--- |
| 项目根目录存在 `.backend-port` 文件 | 文件内容为后端实际端口号（如 `18000`） |
| 浏览器页面左下角连接状态 | 显示绿色"已连接" |
| 连接 Android 设备后设备列表 | 自动显示设备卡片（型号、版本、状态） |
| 点击设备"投屏"按钮 | 跳转至投屏页面，显示设备实时画面 |
| 鼠标在投屏画面上点击/滑动 | 设备响应触控操作 |

> 开发完成后的完整验证流程请参考 [CLAUDE.md](../CLAUDE.md) 中的"开发完成验证清单"，操作记录写入 [memory/ai_step.md](../memory/ai_step.md)。

## 5. 构建与打包

所有构建产物统一输出到 `dist/` 目录：

```
dist/
├── backend/           # Python 后端编译产物
│   └── msca-backend.exe
├── web/               # Vue 前端静态文件（Web 部署用）
│   ├── index.html
│   └── assets/
└── electron/          # Electron 桌面端安装包
    └── MSCA-Setup.exe
```

### 5.1 前端构建

```bash
# 仅构建前端（输出到 frontend/dist/）
npm run build

# 构建并复制到 dist/web/（Web 部署用）
npm run build:web
```

预期结果：生成 `frontend/dist/` 目录，无报错。`build:web` 额外复制到 `dist/web/`。

### 5.2 后端编译（Nuitka）

将 Python 后端编译为独立可执行文件：

```bash
npm run backend:build
```

该脚本会：
1. 在 `backend/` 目录执行 Nuitka 编译
2. 将产物移动到 `dist/backend/msca-backend.exe`
3. 同时复制一份到 `resources/`（供 Electron 打包使用）

首次编译会自动安装 Nuitka 依赖（`nuitka`、`ordered-set`）。

### 5.3 验证后端编译产物

```bash
npm run backend:verify
```

该脚本会自动：
1. 查找 `dist/backend/msca-backend.exe`（回退到 `resources/msca-backend.exe`）
2. 启动 exe 并等待健康检查通过
3. 验证 `/health` 和 `/api/devices` 接口
4. 输出验证结果并自动清理进程

预期输出：`✅ 后端打包验证通过！`

### 5.4 完整打包（Electron 桌面端安装包）

```bash
npm run electron:build
```

该命令依次执行：
1. `npm run build` — 构建 Vue 前端
2. `npm run backend:build` — Nuitka 编译后端
3. `electron-builder` — 打包 Electron 应用

打包内容包括：
- Vue 前端静态文件（`frontend/dist/`）
- Electron 主进程代码（`electron/`）
- 后端可执行文件（`resources/msca-backend.exe`）
- scrcpy-server（`bin/android/`）
- go-ios 工具（`bin/ios/`）

预期结果：生成 `dist/electron/MSCA-Setup.exe` 安装包。

### 5.5 Web 端部署

Web 端需要独立部署前端和后端：

```bash
# 1. 构建前端静态文件
npm run build:web

# 2. 编译后端（或直接用 Python 源码运行）
npm run backend:build
```

部署步骤：
1. 将 `dist/web/` 部署到 Web 服务器（Nginx 等）
2. 将 `dist/backend/msca-backend.exe` 部署到后端服务器（或用 `uv run python __main__.py`）
3. 配置 Nginx 反向代理 API 和 WebSocket

Nginx 参考配置：

```nginx
server {
    listen 443 ssl;
    server_name msca.example.com;

    location / {
        root /path/to/dist/web;
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:18000;
    }

    location /ws/ {
        proxy_pass http://127.0.0.1:18000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400;
    }
}
```

### 5.6 清理构建产物

```bash
npm run dist:clean
```

删除整个 `dist/` 目录。

### 5.7 命令速查

| 命令 | 说明 | 产物位置 |
| :--- | :--- | :--- |
| `npm run build` | 构建 Vue 前端 | `frontend/dist/` |
| `npm run build:web` | 构建前端并复制到 dist | `dist/web/` |
| `npm run backend:build` | Nuitka 编译后端 | `dist/backend/msca-backend.exe` |
| `npm run backend:verify` | 验证后端 exe | — |
| `npm run electron:build` | 完整打包 Electron | `dist/electron/MSCA-Setup.exe` |
| `npm run dist:clean` | 清理所有构建产物 | — |
| `npm run electron:dev` | 桌面端开发模式 | — |
| `npm run dev:backend` | 启动后端开发服务 | — |
| `npm run dev` | 启动前端开发服务 | — |

## 6. iOS 设备连接

### 6.1 前置依赖

| 工具 | 适用版本 | 安装方式 |
| :--- | :--- | :--- |
| tidevice | iOS ≤15.x | `cd backend && uv add tidevice` |
| go-ios | iOS ≥16.x | 已内置于 `bin/ios/ios.exe` |

系统会根据设备 iOS 版本自动选择适配器：
- iOS ≤15.x → tidevice
- iOS ≥16.x → go-ios

### 6.2 连接 iOS 设备

1. 使用 USB 数据线连接 iPhone/iPad 与电脑
2. 设备弹出"信任此电脑"对话框，点击"信任"
3. 验证连接：

```bash
# 使用 go-ios（项目内置）
bin/ios/ios.exe list

# 使用 tidevice（需安装）
cd backend && uv run tidevice list
```

### 6.3 设备配对凭证

首次连接 iOS 设备时，系统会自动生成配对凭证文件 `backend/selfIdentity.plist`，用于建立设备信任关系。

**注意事项**：
- 该文件包含配对密钥，属于敏感信息，已加入 `.gitignore`
- 每台电脑与 iOS 设备首次配对时自动生成
- 删除后需重新信任设备才能重新生成

### 6.4 WDA 签名要求

iOS 投屏依赖 WebDriverAgent（WDA），需要有效的开发者证书签名：

- 免费开发者账号：WDA 签名有效期 7 天，过期需重新签名
- 付费开发者账号（推荐）：签名有效期 1 年

WDA 的 `.ipa` 文件存放于 `resources/` 目录。

### 6.5 iOS 设备调试

```bash
# 查看设备详细信息
bin/ios/ios.exe info --udid=<设备UDID>

# 查看设备 iOS 版本
bin/ios/ios.exe info --udid=<设备UDID> | python -c "import sys,json; print(json.loads(sys.stdin.readline()).get('ProductVersion'))"
```

## 7. 多设备投屏与同步操作

### 7.1 多设备投屏

1. 在设备列表页（首页）连接多台设备
2. 点击设备卡片上的"投屏"按钮，跳转到投屏页面
3. 在投屏页面点击"添加设备"按钮，追加更多设备到网格
4. 网格布局自动适应设备数量（1~4 列可手动切换）

### 7.2 同步操作模式

开启同步模式后，在任一设备上的操作会同时广播到所有投屏设备：

1. 在投屏页面顶部工具栏点击"同步模式"开关
2. 开启后，以下操作会同步到所有设备：
   - 触控操作（点击、滑动）— 自动按比例转换坐标
   - 控制栏按钮（Home、Back、音量、电源等）
   - 文本输入
   - 剪贴板操作

触控同步使用归一化坐标（0~1），不同分辨率设备间自动转换，无需手动适配。

### 7.3 设备名称显示

投屏面板标题栏显示设备名称，优先级：
1. 用户设置的别名（在设置页面配置）
2. 设备型号名称
3. 设备 ID 截断显示

## 8. 常见问题

### Q: `adb devices` 无输出

- 检查 USB 线是否支持数据传输（部分线只能充电）
- 检查手机 USB 调试是否开启
- Windows：检查设备管理器中是否有未识别设备，安装对应驱动
- 执行 `adb kill-server && adb start-server` 重启 ADB 服务

### Q: 后端启动报错 `ModuleNotFoundError`

```bash
cd backend
uv sync  # 重新安装依赖
```

### Q: Electron 启动后白屏

- 使用 `npm run electron:dev` 一键启动即可，它会自动启动前端和后端
- 如果仍然白屏，检查终端输出是否有报错信息

### Q: 端口被占用

后端会自动从 18000 开始探测可用端口，无需手动处理。如需手动指定：

```bash
cd backend
uv run python __main__.py --port 19000
```

### Q: 前端连不上后端（左下角显示"未连接"）

1. 确认后端已启动且 `.backend-port` 文件存在
2. 确认启动顺序：先后端，再前端
3. 如果顺序反了，重启前端即可（Ctrl+C 后重新 `npm run dev`）

### Q: iOS 设备未显示在设备列表中

1. 确认设备已通过 USB 连接并信任了电脑
2. 检查 go-ios 是否能识别设备：`bin/ios/ios.exe list`
3. 如果是 iOS ≤15.x 设备，确认 tidevice 已安装：`cd backend && uv add tidevice`
4. 检查后端日志中是否有 iOS 适配器相关错误

### Q: iOS 投屏黑屏或无画面

1. 确认 WDA 已正确签名并安装到设备
2. 检查 WDA 服务是否正常运行：访问 `http://127.0.0.1:8100/status`
3. 如果端口不通，检查端口转发是否建立成功
