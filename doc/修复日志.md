## 2026-06-11 — 第三轮：打包链路验证与构建修复（Linux 环境）

第二轮唯一待办（Electron 打包模式端口回读的端到端确认）已在 Linux 环境完成，并顺带修复了构建链路上的 2 个问题。

### 1. Nuitka 4.0.8 不兼容 Python 3.14（构建阻塞）

- **位置**：`backend/pyproject.toml`、`backend/uv.lock`
- **根因**：锁定的 Nuitka 4.0.8 的 C 头文件与 CPython 3.14 的 GC 内部结构（`_gc_runtime_state`）不匹配，Scons 后端编译必败；且 `uv run` 隐式同步会把手动升级的版本降回锁定版本。
- **修复**：依赖约束升至 `nuitka>=4.1.2`（官方支持 3.14），用官方 PyPI 源重新生成 uv.lock（最小变更）。
- **注意**：本机 uv 镜像配置会把整个 uv.lock 的源 URL 重写为阿里云镜像，重新生成锁文件时需显式指定 `UV_DEFAULT_INDEX=https://pypi.org/simple`。

### 2. build-backend.mjs 在 Linux 下产物名错配（构建阻塞）

- **位置**：`scripts/build-backend.mjs`
- **根因**：脚本硬编码 `--output-filename=msca-backend.exe`，Linux 产物也带 `.exe`；而脚本收尾查找、`verify-backend.mjs`、`electron/backend-manager.js` 均按平台期望 Linux 下为 `msca-backend`（无后缀），导致"编译成功但报未找到产物"。
- **修复**：输出文件名按平台区分（win32 → `msca-backend.exe`，其他 → `msca-backend`）。

### 3. 打包模式端口回读 E2E 验证（第二轮待办，通过）

- **方法**：真实 `BackendManager`（`isPackaged=true`，stub 替代 electron 模块提供 `app.getPath("userData")`）+ 真实 Nuitka 编译产物；预先占用 18000 并强制探测返回 18000，复现"探测通过后端口被抢占"的 TOCTOU 场景。
- **结果**：
  - 后端检测到 18000 被占用，自动改用 18001，端口文件写入 userData 指定路径 ✅
  - `BackendManager` 健康检查回读实际端口 18001，`start()` 返回 18001 ✅
  - `/health` 200、`stop()` 优雅退出（SIGTERM）、端口文件清理 ✅

### 环境限制说明

- 本轮在 Linux 编译，产物为 **Linux 二进制**；验证覆盖端口回读完整逻辑链路。Windows 发布产物仍需在 Windows 机器上跑一次 `npm run backend:build` + `backend:verify` 做最终确认。
- `verify-backend.mjs` 资源检查中 `bin/ios/ios`（Linux 版 go-ios）缺失为环境限制（仓库仅内置 Windows 版 `ios.exe`），其余 4 项资源检查与功能验证（启动产物 → `/health` → `/api/devices` 200 → SIGTERM 优雅退出）全部通过。
- Linux standalone 编译需要 `patchelf`，已通过 PyPI 包（`uv pip install patchelf`）装入项目 venv，无需系统级安装。

## 2026-06-11 — 第二轮：遗留缺陷清零（4 项）

第一轮排查中评估为"工程量大/需实测"而遗留的 4 项缺陷，本轮全部修复。

### 1. 设备旋转后 screen_size 不更新，旋转后 Android 反控失效（高）

- **位置**：`backend/app/scrcpy/server_manager.py`、`backend/app/scrcpy/protocol.py`
- **根因**：屏幕宽高只在握手时从 codec header 读取一次。scrcpy v3 server 会校验触控消息携带的宽高与设备当前显示尺寸一致，不一致则丢弃事件——设备旋转后宽高互换，所有触控被静默丢弃。
- **修复**：
  - `protocol.py` 新增 H.264 SPS 宽高解析（`_BitReader` Exp-Golomb 位读取器 + 防竞争字节剥离 + 裁剪单位计算，支持 Baseline/High profile、场编码、frame cropping）；
  - `server_manager.read_video_frame` 识别 frame meta 中 pts 字段的 `PACKET_FLAG_CONFIG`（bit 63）标志——设备旋转/分辨率变化时 server 会重发配置包（SPS/PPS），仅对配置包解析 SPS 并刷新 `_screen_width/_screen_height`，普通帧零额外开销。
- **测试**：新增 `tests/test_h264_sps.py`（9 个用例，测试内置规范级 SPS 位流写入器交叉验证解析器）+ `tests/test_scrcpy_device_messages.py` 旋转场景 2 个用例。

### 2. 后端端口探测 TOCTOU 与 --host 不一致（中）

- **位置**：`backend/__main__.py`
- **根因**：①探测 socket 关闭后到 uvicorn 实际 bind 之间端口可能被抢占，启动失败但 `.backend-port` 已写入旧值；②探测固定绑 `127.0.0.1`，传 `--host 0.0.0.0` 时探测结论不代表实际可绑定。
- **修复**：探测地址改为与 `args.host` 一致；uvicorn 启动失败（`SystemExit` 退出码 3，即 `STARTUP_FAILURE`）时清理端口文件并自动尝试下一个端口，最多 10 个。
- **测试**：新增 `tests/test_backend_entry.py`（3 个用例）；实测占用 18000 后启动，后端自动改用 18001 且健康检查通过、退出后端口文件正确清理。

### 3. parse_nal_units 起始码回退计算错误（低，潜伏）

- **位置**：`backend/app/scrcpy/protocol.py`
- **根因**：对非开头位置的 3 字节起始码（`00 00 01`），切片回退按位置而非实际起始码长度判断，返回的 NAL 数据会多带前一个无关字节。当前调用方只消费 `nal_type`（单独计算，结果正确）故未爆发；本轮新增的 SPS 解析开始消费 NAL 数据，必须先修。
- **修复**：扫描统一按 `00 00 01` 匹配并记录前缀实际起始位置（前一字节为 `00` 即 4 字节起始码），单元边界按实际前缀长度回退。
- **测试**：`tests/test_h264_sps.py` 中混合 3/4 字节起始码边界用例 2 个。

### 4. Electron 打包模式端口回读缺失（中）

- **位置**：`electron/backend-manager.js`
- **根因**：第一轮只修了 dev 模式从 `.backend-port` 回读实际端口；打包模式因 Nuitka 产物的端口文件默认落点不可靠而未实现，端口被抢占时健康检查会盯错端口 15 秒后误判启动失败。
- **修复**：两种模式统一通过 `--port-file` 显式指定端口文件路径——打包模式写到 `userData` 目录（resources 目录在 Program Files 安装时只读），dev 模式仍为项目根目录；`_refreshPortFromFile` 去掉打包模式守卫，spawn 前统一清理残留端口文件。
- **备注**：打包链路的端到端确认需在下次执行 `npm run backend:build` + `backend:verify` 时覆盖。
