  - `--userspace` 模式（MSCA 首选）不依赖 wintun 也不需要管理员；wintun 仅影响提权内核模式
  - wintun 预编译二进制许可允许"随仅通过其 API 使用该软件的软件"再分发（go-ios 即此用法），已附 `bin/ios/wintun-LICENSE.txt`
- **修复**：
  - 内嵌 wintun 0.14.1（amd64）到 `bin/ios/`，与 ios.exe 同目录，打包零配置生效
  - `goios_adapter` 新增 `_wintun_available()`：提权内核模式前检测（exe 同目录 + System32），缺失时输出明确告警与指引；tunnel 最终失败的错误信息附 wintun 提示
  - `ios-tunnel.bat` 启动前检查 wintun.dll，缺失时自动从 `bin/ios/` 复制到 System32（脚本本身已是管理员）
- **文档**：新增 [设备部署指南](设备部署指南.md)（WDA 打包签名 + Android 设备准备），操作手册/dist/README 排障项补 wintun 说明

## 2026-06-12 — 第六轮：关闭应用后子进程残留修复

用户在 Windows 打包验证时发现：上次启动的 MSCA 留下 `ios.exe` 和 `msca-backend.exe` 进程，导致 electron-builder 清理 `win-unpacked` 时文件被占用拒绝。

### 1. go-ios tunnel agent（ios.exe）无人负责清理

- **位置**：`backend/app/drivers/adapters/goios_adapter.py`、`backend/app/main.py`
- **根因**：tunnel agent 是主机级共享守护进程（多台 iOS 17+ 设备共用），设备级 `stop_wda()` 刻意不杀它——但应用关闭路径（lifespan shutdown）也没有任何代码负责终止它，因此**必然残留**。
- **修复**：新增模块级共享 agent 进程注册表，`_try_tunnel_start()` 启动成功时登记；新增 `shutdown_tunnel_agents()` 统一终止（terminate → 3s 超时 → kill），在 `main.py` lifespan 关闭流程末尾调用。提权（UAC）方式启动的 agent 为独立守护进程、无句柄，不在清理范围（属于用户手动管理）。

### 2. Windows 上 Electron 停止后端不杀子进程树

- **位置**：`electron/backend-manager.js` `_killProcess()`
- **根因**：Windows 上 Node 的 `proc.kill("SIGTERM")` 等价于 `TerminateProcess`——**强杀且只杀后端自身**：后端 lifespan 清理没有机会执行，其子进程（ios.exe、端口转发等）全部变成孤儿残留。
- **修复**：Windows 分支改用 `taskkill /PID <pid> /T /F` 整树强制终止（等待 exit 事件，5s 超时；taskkill 失败时回退 `SIGKILL`）；POSIX 保持原 SIGTERM 优雅退出 → 5s 超时 SIGKILL 流程（后端可走 lifespan 清理自己的子进程）。
- **效果**：双保险——正常退出时后端 lifespan 主动清理 agent（第 1 项），Windows 强杀场景由 taskkill 整树兜底,关闭 MSCA 后不再残留 `msca-backend.exe`/`ios.exe`。

### 验证

- 新增 4 项单元测试（agent 注册清理、已退出跳过、terminate 失败回退 kill、空注册表幂等），后端 57 项测试全部通过，ruff 通过
- Electron 6 项单测通过；backend-manager.js 仅存量 `while(true)` lint 告警（非本次引入）
- 实测：后端启动 → `/health` ok → SIGTERM → lifespan 清理完整执行、端口文件删除、无残留进程
- Windows 真实场景（关闭 MSCA 后任务管理器无 ios.exe/msca-backend.exe）待用户回归

## 2026-06-12 — 第五轮：iOS 真机触控坐标修复与后端运行日志

本轮针对 N1 真机回归发现的问题：iOS 17.7.4 无法点击、iOS 15.1 触控位置与页面响应位置不一致（快捷键正常），并补充用户可见的后端日志查看入口。

### 1. iOS 触控坐标系错配（像素 vs 点）— 两台设备问题的共同根因

- **位置**：`backend/app/drivers/ios.py`、`backend/app/websocket/control.py`、（坐标来源）`frontend/src/composables/useDeviceControl.js`
- **根因**：前端把鼠标位置换算为 **MJPEG 帧像素坐标**（`videoWidth/videoHeight` 为解码帧像素尺寸）发给后端；后端 iOS 驱动将其**原样**写入 WDA W3C Actions——但 WDA Actions 坐标系是**点（point，逻辑分辨率）**。iPhone scale 为 2x/3x，帧像素 = 点 × scale，因此发往 WDA 的坐标被系统性放大 2~3 倍：
  - **15.1 设备**：点击/滑动位置整体偏移（仅左上 1/scale 区域能落回屏幕内）→ "实际触控与页面响应位置不一致"
  - **17.7.4 设备**：tap 落点越界被 WDA 忽略 → "无法点击"；swipe 被钳制后相对位移仍能触发滚动 → "滑动正常"的表象
  - **快捷键正常**：home/lock/volume 走 `pressButton`/`homescreen` 端点，不涉及坐标，故不受影响
- **修复**：
  - `ios.py` 启动投屏时分别记录**窗口点尺寸**（`/window/size`，WDA 目标坐标系）与**帧像素尺寸**（截图探测），并打印 scale 日志便于排障
  - 新增 `_to_window_points()`：按 `点 = 像素 × 窗口/帧` 比例换算并钳制到窗口范围；`send_event()` 的 tap/swipe 坐标统一经此换算
  - `control.py` iOS 分支把前端随事件携带的 `width/height`（帧尺寸）透传到 ControlEvent（touch down 时暂存、up 时并入 tap/swipe 参数），旋转后帧尺寸变化仍跟手；窗口尺寸不可用时坐标原样透传（行为与修复前一致）
- **验证**：新增 6 项单元测试（2x/3x 换算、事件帧尺寸回退、透传兜底、聚合事件携带帧尺寸），后端 53 项测试全部通过；真机回归待 Windows 端执行

### 2. 后端运行日志用户不可见

- **位置**：`backend/app/core/log_buffer.py`（新增）、`backend/app/api/logs.py`（新增）、`frontend/src/views/LogsView.vue`（新增）、`electron/backend-manager.js`
- **根因**：后端日志只输出到 stdout，由 Electron 主进程 console 吞掉，用户在桌面端无法查看设备连接/控制失败等关键日志。
- **修复**：
  - 后端 logging 增加内存环形缓冲（最近 2000 条）+ RotatingFileHandler 落盘（`logs/backend.log`，5MB×3 滚动）；`GET /api/logs?lines=` 返回最近日志与日志文件路径
  - 打包模式由 BackendManager 传 `--log-dir`（userData/logs，安装目录可能只读）；dev 模式默认 `backend/logs/`（已在 .gitignore）
  - 前端新增"运行日志"侧边栏页面：2 秒自动刷新（可关）、底部跟随滚动、一键复制，桌面端与 Web 端通用
- **验证**：后端实测 `/health` 与 `/api/logs` 返回正常、`backend/logs/backend.log` 落盘正常；前端构建通过、14 项前端单测与 2 项 Electron 单测通过

## 2026-06-12 — 第四轮：桌面版打包规则确立与阻塞问题修复

本轮在 Windows 环境执行桌面版完整打包，确认并记录打包链路的必要条件与关键规则。

### 1. electron-builder 要求 ICO 包含 ≥256×256 帧

- **位置**：`resources/icon.ico`
- **根因**：`package.json` 中 `win.icon` 指向 `resources/icon.ico`，但该文件不存在；且 electron-builder 25.x 对 ICO 文件严格校验，要求包含 256×256 尺寸的帧，否则报"image must be at least 256x256"。
- **修复**：用 Pillow `struct.pack` 手动构建 ICO 二进制，包含 16/32/48/256px 四个帧，256 帧以 PNG 压缩格式内嵌（直接以 `struct.pack("<BBBBHHII", 0, 0, ...)` 写入 256 帧宽高为 0 的 ICONDIRENTRY，符合 ICO 规范对 256 尺寸的表示方式）。存入 `resources/icon.ico`，SVG 原件存档 `resources/icon.svg`。
- **规则**：更换图标时，替换 `resources/icon.svg` 后必须重新生成 `resources/icon.ico`（256 帧 PNG 压缩，宽高字段写 0）。

### 2. 打包前后端目录必须为 standalone 结构

- **位置**：`resources/msca-backend/`
- **根因**：`electron/backend-manager.js` 生产路径为 `process.resourcesPath/resources/msca-backend/msca-backend.exe`（standalone 目录），但此前 `resources/` 下只存放了裸可执行文件（非目录结构），导致安装包内嵌后端路径不对。
- **修复**：执行 `npm run backend:build`，Nuitka standalone 完整目录会被复制到 `resources/msca-backend/`。每次打包前必须确认该目录存在且包含完整运行时文件。
- **规则**：不能手动复制单个 `.exe`，必须通过 `npm run backend:build` 生成完整 standalone 目录。

### 打包命令参考

```bash
# 全量打包（含前端 build + Nuitka 编译，约 15 分钟）
npm run electron:build

# 仅打包（前端与后端产物已是最新时使用，节省时间）
cross-env ELECTRON_BUILDER_BINARIES_MIRROR=https://npmmirror.com/mirrors/electron-builder-binaries/ npx electron-builder
```

## 2026-06-11 — 第三轮：打包链路验证与构建修复（Linux 环境）

第二轮唯一待办（Electron 打包模式端口回读的端到端确认）已在 Linux 环境完成，并顺带修复了构建链路上的 2 个问题。

### 1. Nuitka 4.0.8 不兼容 Python 3.14（构建阻塞）

- **位置**：`backend/pyproject.toml`、`backend/uv.lock`
- **根因**：项目最低 Python 版本为 3.13（`requires-python = ">=3.13"`，未变更），但本机 venv 实际为 Python 3.14.5；锁定的 Nuitka 4.0.8 的 C 头文件与 CPython 3.14 的 GC 内部结构（`_gc_runtime_state`）不匹配，Scons 后端编译必败。且 `uv run` 隐式同步会把手动升级的版本降回锁定版本。
- **修复**：依赖约束升至 `nuitka>=4.1.2`（同时兼容 Python 3.13/3.14），用官方 PyPI 源重新生成 uv.lock（最小变更）。对 3.13 用户无影响。
- **注意**：本机 uv 镜像配置会把整个 uv.lock 的源 URL 重写为阿里云镜像，重新生成锁文件时需显式指定 `UV_DEFAULT_INDEX=https://pypi.org/simple`。

### 2. build-backend.mjs 在 Linux 下产物名错配（构建阻塞）

- **位置**：`scripts/build-backend.mjs`
- **根因**：脚本硬编码 `--output-filename=msca-backend.exe`，Linux 产物也带 `.exe`；而脚本收尾查找、`verify-backend.mjs`、`electron/backend-manager.js` 均按平台期望 Linux 下为 `msca-backend`（无后缀），导致"编译成功但报未找到产物"。
- **修复**：输出文件名按平台区分（win32 → `msca-backend.exe`，其他 → `msca-backend`）。

### 3. 打包模式端口回读 E2E 验证（第二轮待办，通过）

- **方法**：真实 `BackendManager`（`isPackaged=true`，stub 替代 electron 模块提供 `app.getPath("userData")`）+ 真实 Nuitka 编译产物；预先占用 18000 并强制探测返回 18000，复现"探测通过后端口被抢占"的 TOCTOU 场景。
- **结果**：

  - `BackendManager` 健康检查回读实际端口 18001，`start()` 返回 18001 ✅
  - `/health` 200、`stop()` 优雅退出（SIGTERM）、端口文件清理 ✅

## 2026-06-11 — 第二轮：遗留缺陷清零（4 项）

第一轮排查中评估为"工程量大/需实测"而遗留的 4 项缺陷，本轮全部修复。

### 1. 设备旋转后 screen_size 不更新，旋转后 Android 反控失效（高）

- **位置**：`backend/app/scrcpy/server_manager.py`、`backend/app/scrcpy/protocol.py`
- **根因**：屏幕宽高只在握手时从 codec header 读取一次。scrcpy v3 server 会校验触控消息携带的宽高与设备当前显示尺寸一致，不一致则丢弃事件——设备旋转后宽高互换，所有触控被静默丢弃。
- **修复**：
  - `protocol.py` 新增 H.264 SPS 宽高解析（`_BitReader` Exp-Golomb 位读取器 + 防竞争字节剥离 + 裁剪单位计算，支持 Baseline/High profile、场编码、frame cropping）；
  - `server_manager.read_video_frame` 识别 frame meta 中 pts 字段的 `PACKET_FLAG_CONFIG`（bit 63）标志——设备旋转/分辨率变化时 server 会重发配置包（SPS/PPS），仅对配置包解析 SPS 并刷新 `_screen_width/_screen_height`，普通帧零额外开销。
- **测试**：新增 `tests/test_h264_sps.py`（9 个用例，测试内置规范级 SPS 位流写入器交叉验证解析器）+ `tests/test_scrcpy_device_messages.py` 旋转场景 2 个用例。

### 2. 后端端口探测 TOCTOU 与 --host 不一致（中）

- **位置**：`backend/__main__.py`
- **根因**：①探测 socket 关闭后到 uvicorn 实际 bind 之间端口可能被抢占，启动失败但 `.backend-port` 已写入旧值；②探测固定绑 `127.0.0.1`，传 `--host 0.0.0.0` 时探测结论不代表实际可绑定。
- **修复**：探测地址改为与 `args.host` 一致；uvicorn 启动失败（`SystemExit` 退出码 3，即 `STARTUP_FAILURE`）时清理端口文件并自动尝试下一个端口，最多 10 个。
- **测试**：新增 `tests/test_backend_entry.py`（3 个用例）；实测占用 18000 后启动，后端自动改用 18001 且健康检查通过、退出后端口文件正确清理。

### 3. parse_nal_units 起始码回退计算错误（低，潜伏）

- **位置**：`backend/app/scrcpy/protocol.py`
- **根因**：对非开头位置的 3 字节起始码（`00 00 01`），切片回退按位置而非实际起始码长度判断，返回的 NAL 数据会多带前一个无关字节。当前调用方只消费 `nal_type`（单独计算，结果正确）故未爆发；本轮新增的 SPS 解析开始消费 NAL 数据，必须先修。
- **修复**：扫描统一按 `00 00 01` 匹配并记录前缀实际起始位置（前一字节为 `00` 即 4 字节起始码），单元边界按实际前缀长度回退。
- **测试**：`tests/test_h264_sps.py` 中混合 3/4 字节起始码边界用例 2 个。

### 4. Electron 打包模式端口回读缺失（中）

- **位置**：`electron/backend-manager.js`
- **根因**：第一轮只修了 dev 模式从 `.backend-port` 回读实际端口；打包模式因 Nuitka 产物的端口文件默认落点不可靠而未实现，端口被抢占时健康检查会盯错端口 15 秒后误判启动失败。
- **修复**：两种模式统一通过 `--port-file` 显式指定端口文件路径——打包模式写到 `userData` 目录（resources 目录在 Program Files 安装时只读），dev 模式仍为项目根目录；`_refreshPortFromFile` 去掉打包模式守卫，spawn 前统一清理残留端口文件。
- **备注**：打包链路的端到端确认需在下次执行 `npm run backend:build` + `backend:verify` 时覆盖。
