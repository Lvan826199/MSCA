## 2. Android 核心技术点

### 2.1 Scrcpy 协议与服务器生命周期

Scrcpy 是 Android 投屏的核心，通过 ADB 在设备上启动 `scrcpy-server.jar`，建立视频流和控制通道。

**启动流程**（`ScrcpyServerManager`）：

1. 通过 ADB 推送 `scrcpy-server.jar` 到设备 `/data/local/tmp/`
2. 创建 ADB 端口转发：`localabstract:scrcpy`
3. 通过 `app_process` 启动 scrcpy-server，传入参数：
   - `max_size`：视频最大分辨率
   - `max_fps`：最大帧率
   - `bitrate`：视频码率
   - `video_codec=h264`：编码格式
4. 建立两个 socket 连接：
   - **Video Socket**：接收 H.264 视频流
   - **Control Socket**：双向控制通道

**握手协议**（scrcpy 3.3+）：

1. 连接 Video Socket → 读取 1 字节 dummy byte（服务器就绪信号）
2. 连接 Control Socket
3. 服务器发送设备元数据：
   - `device_name`：64 字节
   - `codec_header`：12 字节（codec_id 4B + width 4B + height 4B）

### 2.2 H.264 视频流处理

**帧读取**（后端）：

- 每帧元数据：12 字节（pts 8B + packet_size 4B，大端序）
- 帧数据：原始 H.264 NAL 单元（Annex B 格式，带 start code）
- 非阻塞读取：0.1s 超时

**NAL 单元类型**（`protocol.py`）：

| NAL Type | 值 | 说明 |
|----------|---|------|
| SPS | 7 | 序列参数集（解码器配置） |
| PPS | 8 | 图像参数集（解码器配置） |
| IDR | 5 | 关键帧（I 帧） |
| SLICE | 1 | 非关键帧（P/B 帧） |
| SEI | 6 | 补充增强信息 |
| AUD | 9 | 访问单元分隔符 |

**WebSocket 传输协议**：

- 首帧：JSON 配置 `{type: "config", width, height, codec: "h264"}`
- 后续帧：1 字节帧类型头 + H.264 数据
  - `0x01` = 关键帧（含 SPS/PPS/IDR）
  - `0x00` = 非关键帧（P/B 帧）
- SPS/PPS 缓存：后端缓存配置数据，等 IDR 帧到达后合并发送

### 2.3 前端 H.264 解码（WebCodecs API）

**Annex B → AVC 格式转换**（`useVideoDecoder.js`）：

浏览器 WebCodecs API 要求 AVC 格式（ISO 14496-15），而 scrcpy 输出 Annex B 格式：

- **Annex B**：NAL 单元以 start code（`0x000001` 或 `0x00000001`）分隔
- **AVC**：NAL 单元以 4 字节大端长度前缀分隔，不含 SPS/PPS

转换步骤：
1. `parseNalUnits()`：定位 start code，提取 NAL 边界和类型
2. `buildAvcConfig()`：从首个关键帧提取 SPS/PPS，构建 AVCDecoderConfigurationRecord
3. `annexBToAvc()`：剥离非 VCL NAL（SPS/PPS/AUD/SEI），将 start code 替换为长度前缀

**AVCDecoderConfigurationRecord 结构**：

```
configurationVersion(1B) + profile(1B) + compatibility(1B) + level(1B)
+ lengthSizeMinusOne(1B) + numSPS(1B) + SPS_length(2B) + SPS_data
+ numPPS(1B) + PPS_length(2B) + PPS_data
```

**解码器配置**：

```javascript
decoder.configure({
  codec: "avc1.640028",        // 从 SPS 提取的 profile/level
  optimizeForLatency: true,    // 低延迟模式
  description: avcConfigBuffer // AVCDecoderConfigurationRecord
})
```

**屏幕旋转检测**：通过 SPS 变化检测分辨率改变，自动重新配置解码器。

### 2.4 Scrcpy 控制协议

**二进制编码格式**（`protocol.py`）：

| 指令类型 | 编码结构 |
|---------|---------|
| 触摸 | type(1B) + action(1B) + pointerId(8B) + x(4B) + y(4B) + width(2B) + height(2B) + pressure(2B) + actionButton(4B) + buttons(4B) |
| 按键 | type(1B) + action(1B) + keycode(4B) + repeat(4B) + metastate(4B) |
| 文本 | type(1B) + length(4B) + utf8_text |
| 滚动 | type(1B) + x(4B) + y(4B) + hScroll(4B) + vScroll(4B) + buttons(4B) |

**前端坐标映射**（`useDeviceControl.js`）：

```
Canvas 显示坐标 → 设备实际坐标
canvasToDevice(): 根据 canvas 显示区域与视频实际分辨率的比例进行换算
坐标 clamp 到设备边界范围内
```

**键盘映射**：Browser `KeyboardEvent.code` → Android keycode（A-Z: 29-54, 0-9: 7-16, 功能键、方向键、符号键等）

## 3. iOS 核心技术点

### 3.1 WDA 双端口架构

WebDriverAgent 在设备上运行两个独立的 HTTP 服务：

| 服务 | 设备端口 | 功能 |
|------|---------|------|
| WDA REST API | 8100 | 设备控制（点击、滑动）、截图、session 管理 |
| MJPEG Server | 9100 | 视频流服务，输出连续 JPEG 帧 |

**关键认知**：这两个端口必须分别转发到本地，不能只转发 8100 然后从 8100 上请求 MJPEG 流（会 404）。

### 3.2 WDA Bundle ID 自动检测

**问题背景**：不同来源的 WDA 有不同的 bundle ID：
- Facebook 官方：`com.facebook.WebDriverAgentRunner.xctrunner`
- AScript-WDA：`com.ascript.webdriveragentrunner.xctrunner`
- Appium WDA：`com.appium.WebDriverAgentRunner.xctrunner`

**解决方案**（参考 tidevice/Airtest 的 fnmatch 方式）：

1. 读取配置文件 `wda_config.json`，如有手动指定则直接使用
2. 否则用 `fnmatch` 模糊匹配模式 `com.*.xctrunner` 遍历设备已安装 app
3. 匹配到多个时，优先选择非 facebook 的（用户自定义签名的 WDA）
4. go-ios 通过 `ios apps` 命令获取 app 列表
5. tidevice 通过 `td.installation.iter_installed()` 获取

**配置文件**（`backend/config/wda_config.json`）：

```json
{
  "wda_bundle_id": "",                          // 手动指定，为空则自动检测
  "wda_bundle_id_pattern": "com.*.xctrunner",   // 模糊匹配模式
  "mjpeg_port_on_device": 9100,                 // 设备端 MJPEG 端口
  "wda_port_on_device": 8100                    // 设备端 WDA API 端口
}
```

### 3.3 iOS 版本适配策略

| iOS 版本 | 适配器 | 工具 | 端口转发命令 | WDA 启动命令 |
|----------|--------|------|-------------|-------------|
| ≤15.x | TideviceAdapter | tidevice (Python) | `tidevice relay <local> <device>` | `tidevice wdaproxy -p <port> -B <bundleid>` |
| ≥16.x | GoIOSAdapter | go-ios (CLI) | `ios forward <local> <device>` | `ios runwda --bundleid=<id> --testbundleid=<id>` |
| ≥17.x | GoIOSAdapter | go-ios (CLI) | 同上 + tunnel agent | 需先 `ios tunnel start [--userspace]` |

### 3.4 WDA 启动策略（两阶段探测）

```
阶段 1：Relay 模式（快速路径）
  ├─ 启动端口转发（WDA API 端口）
  ├─ 轮询 /status 检查 WDA 是否已在设备上运行
  ├─ 如果健康 → 启动 MJPEG 端口转发 → 完成
  └─ 如果超时 → 进入阶段 2

阶段 2：完整启动模式
  ├─ 自动检测 WDA bundle ID
  ├─ 启动 WDA API 端口转发
  ├─ 启动 wdaproxy/runwda（传入 bundle ID）
  ├─ 轮询 /status 等待 WDA 就绪（最多 30s）
  ├─ 启动 MJPEG 端口转发
  └─ 完成
```

**Relay 模式的意义**：用户可能已经通过其他工具（如 Xcode、Appium）启动了 WDA，此时只需做端口转发即可，无需重复启动。

### 3.5 MJPEG 流处理

**后端读取**（`IOSDriver._read_mjpeg_loop`）：

1. 连接 MJPEG 端口（独立于 WDA API 端口）
2. 持续读取 HTTP 响应体的原始字节流
3. 在 buffer 中搜索 JPEG 帧边界：
   - 帧起始：`0xFFD8`（JPEG SOI 标记）
   - 帧结束：`0xFFD9`（JPEG EOI 标记）
4. 提取完整 JPEG 帧，通过 `asyncio.Queue` 分发给 WebSocket 订阅者
5. 队列满时丢弃最旧帧，保证实时性

**前端解码**：

```javascript
// iOS MJPEG 解码（比 H.264 简单得多）
const blob = new Blob([jpegData], { type: 'image/jpeg' })
const bitmap = await createImageBitmap(blob)
ctx.drawImage(bitmap, 0, 0, canvas.width, canvas.height)
```

**WebSocket 传输**：
- 首帧：JSON 配置 `{type: "config", width, height, codec: "mjpeg"}`
- 后续帧：原始 JPEG 二进制数据
- 心跳：JSON `{type: "ping"}` 每 5 秒

### 3.6 iOS 控制指令（WDA REST API）

| 操作 | WDA 端点 | 请求体 |
|------|---------|--------|
| 点击 | `POST /session/{id}/wda/tap/0` | `{x, y}` |
| 滑动 | `POST /session/{id}/wda/dragfromtoforduration` | `{fromX, fromY, toX, toY, duration}` |
| Home 键 | `POST /wda/homescreen` | — |
| 锁屏 | `POST /wda/lock` | — |

### 3.7 go-ios Tunnel Agent（iOS 17+）

iOS 17 引入了新的设备通信机制，go-ios 需要先启动 tunnel agent：

```
尝试顺序：
1. ios tunnel start --userspace  (用户态模式，无需管理员权限)
2. ios tunnel start              (默认模式，可能需要管理员权限)

验证方式：ios tunnel ls（检查 tunnel 是否就绪）
超时：每种模式最多等待 15 秒
```

## 4. 通信架构

### 4.1 WebSocket 通道

| 通道 | 路径 | 数据格式 | 方向 |
|------|------|---------|------|
| 视频流 | `/ws/video/{device_id}` | 二进制帧 | 服务端 → 客户端 |
| 控制指令 | `/ws/control/{device_id}` | JSON | 双向 |

### 4.2 视频流 WebSocket 协议

**Android (H.264)**：
```
[首帧] JSON: {type: "config", width: 1080, height: 1920, codec: "h264"}
[后续] Binary: [0x01] + SPS + PPS + IDR  (关键帧)
[后续] Binary: [0x00] + P/B frame data    (非关键帧)
```

**iOS (MJPEG)**：
```
[首帧] JSON: {type: "config", width: 375, height: 812, codec: "mjpeg"}
[后续] Binary: 完整 JPEG 数据 (FFD8...FFD9)
[心跳] JSON: {type: "ping"}
```

### 4.3 控制指令 WebSocket 协议

```json
// 触摸
{"action": "touch", "params": {"action": "down", "x": 540, "y": 960, "width": 1080, "height": 1920}}

// 按键
{"action": "key", "params": {"action": "down", "keycode": 4, "metastate": 0}}

// 文本输入
{"action": "text", "params": {"text": "hello"}}

// 系统按键
{"action": "back"}
{"action": "home"}
```

### 4.4 前端连接模式

| 模式 | 说明 | 后端地址来源 |
|------|------|-------------|
| Electron | 桌面端 | `window.electronAPI.getBackendPort()` → `http://127.0.0.1:{port}` |
| Vite Dev | 开发模式 | 空 backendUrl → 相对路径 → Vite proxy |
| Production Web | 生产 Web | `window.location.origin`（同源部署） |

### 4.5 WebSocket 重连机制

- 心跳：每 30 秒发送 `"ping"`
- 重连：指数退避（1s → 2s → 4s → 8s → 16s → 32s），最多 10 次
- 状态流转：`connecting` → `connected` → `disconnected`

## 5. 端口管理策略

| 端口范围 | 用途 | 分配方式 |
|---------|------|---------|
| 8100, 8110, 8120... | WDA API 端口 | 每设备递增 10，自动跳过已占用端口 |
| 8101, 8111, 8121... | MJPEG 流端口 | WDA 端口 + 1（被占用则 +2） |
| 18000+ | 后端 HTTP/WS 服务 | 自动探测可用端口，写入 `.backend-port` |

**端口分配逻辑**（`IOSDriver._allocate_wda_ports`）：
- 全局计数器递增，基数 8100，步进 10
- 尝试 `socket.bind()` 检测端口是否空闲
- 最多尝试 20 次
- 端口被占用时自动清理（`kill_process_on_port`）

## 6. 多设备并发架构

### 6.1 发布-订阅模型

每个设备驱动维护独立的视频帧订阅列表：

```python
# 后端：多个 WebSocket 客户端订阅同一设备的视频流
queue = driver.subscribe_video()      # 创建 asyncio.Queue
# ... WebSocket 循环中从 queue 取帧发送 ...
driver.unsubscribe_video(queue)       # 断开时取消订阅
```

### 6.2 背压控制

- 视频帧队列 `maxsize=5`（iOS）/ `maxsize=30`（Android）
- 队列满时丢弃最旧帧，保证最新帧优先
- 防止慢客户端导致内存暴涨

### 6.3 多设备同步控制

- 前端 `useDeviceControl.js` 支持同步模式
- 触摸事件归一化为 `{nx, ny}`（0-1 范围）
- 广播到其他面板，`_isSyncReceiving` 标志防止反馈循环

## 9. 问题分析与修复记录

### 问题 1：iOS 15.1 MJPEG 流 404

- **现象**：投屏启动后 MJPEG 流返回 HTTP 404，无画面
- **根因**：WDA 的 MJPEG 视频流服务运行在设备端口 9100（MJPEG Server），而非 WDA API 端口 8100。之前代码只转发了 8100，然后尝试从 8100 上的 `/stream.mjpeg` 获取流，自然返回 404
- **解决方案**：分别转发设备端口 8100（WDA API）和 9100（MJPEG 流）到两个不同的本地端口
- **涉及文件**：`ios.py`、`tidevice_adapter.py`、`goios_adapter.py`、`base.py`

### 问题 2：iOS 16+ WDA bundle ID 写死

- **现象**：go-ios `runwda` 报错 `Did not find test app for 'com.facebook.WebDriverAgentRunner.xctrunner'`
- **根因**：go-ios 默认使用 facebook 的 bundle ID，而用户安装的是 AScript-WDA（`com.ascript.webdriveragentrunner.xctrunner`）。代码没有传 `--bundleid` 参数
- **解决方案**：参考 tidevice/Airtest 的 fnmatch 模糊匹配方式（`com.*.xctrunner`），自动检测设备上已安装的 WDA bundle ID，同时支持配置文件手动指定
- **涉及文件**：`goios_adapter.py`、`tidevice_adapter.py`、`base.py`、`wda_config.json`
